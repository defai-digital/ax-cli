# PRD: `/init` Command Refactor

**Version:** 1.0
**Date:** 2026-01-17
**Author:** AutomatosX Engineering
**Status:** Draft
**Stakeholders:** ax-cli, ax-grok users

---

## Executive Summary

Refactor the `/init` command to simplify output from 3 files to a single `AX.md`, remove opinionated hardcoded paths, unify provider directories, and improve content quality with project-specific instructions derived from analysis.

This PRD is based on a multi-model synthesis discussion (Claude, Grok, Gemini) that reached 70% consensus on key improvements.

---

## Problem Statement

### Current Issues

1. **Multi-file complexity**: The current implementation generates 3 files (`ax.index.json`, `ax.summary.json`, `.ax-{provider}/CUSTOM.md`), creating unnecessary fragmentation and cognitive overhead.

2. **Opinionated paths**: Hardcoded `automatosx/PRD/`, `automatosx/REPORT/`, `automatosx/tmp/` directory structures are baked into `CUSTOM.md`, alienating users with established workflows.

3. **Provider fragmentation**: Separate `.ax-grok/` and `.ax-cli/` directories create synchronization drift when the same project is used with multiple providers.

4. **Generic content**: DO/DON'T sections contain generic advice ("DON'T introduce security vulnerabilities") that any LLM already knows, adding token overhead without value.

5. **Compatibility gap**: Single-file approaches (like Claude Code's `CLAUDE.md`) have become common; our multi-file approach creates friction for users switching between tools.

### Impact

- User adoption friction due to rigid structure requirements
- Token inefficiency from generic boilerplate content
- Maintenance burden from provider-specific directories
- Reduced portability compared to competitors

## User Stories

### US-1: New Project Setup
**As a** developer starting a new project
**I want** to run `ax init` and get a single context file
**So that** I can immediately start working with any AI coding assistant

**Acceptance Criteria:**
- Single `AX.md` generated at project root
- File contains build/test commands detected from project
- No hardcoded directory structures imposed
- Works with ax-grok, ax-cli, and other AI tools

### US-2: Existing Project Migration
**As a** user with existing `.ax-grok/CUSTOM.md` files
**I want** to migrate to the new single-file format
**So that** I don't lose my customizations

**Acceptance Criteria:**
- `ax init --migrate` converts old format to new
- Existing customizations preserved
- Old files optionally cleaned up
- Clear migration output showing what changed

### US-3: Deep Analysis on Demand
**As a** developer on a complex project
**I want** to request detailed architecture/security analysis
**So that** I get comprehensive context when needed

**Acceptance Criteria:**
- `ax init --depth=full` generates Tier 3 analysis
- `ax init --depth=security` includes Tier 4
- Deep analysis stored in `.ax/analysis.json` (gitignored)
- Main `AX.md` stays lean by default

### US-4: Multi-Provider Workflow
**As a** developer using both ax-grok and ax-cli
**I want** a unified configuration
**So that** I don't maintain duplicate settings

**Acceptance Criteria:**
- Single `.ax/` directory for all providers
- `config.json` is provider-agnostic
- Provider-specific hints as inline comments in `AX.md`

## Detailed Requirements

### 1. Output Structure

#### 1.1 Primary Output: `AX.md`

**Location:** Project root
**Commit status:** Should be committed to version control
**Token budget:** 500-800 tokens (default), up to 1500 (with --depth=full)

**Structure:**
```markdown
<!--
Generated by: ax-cli / ax-grok
Last updated: 2026-01-17
Refresh: Run `/init` or `ax init --refresh` to update
-->

# Project Name

**Type:** cli | **Language:** TypeScript | **PM:** pnpm

## Build & Development

```bash
pnpm install    # Install dependencies
pnpm build      # Build all packages
pnpm test       # Run tests
pnpm lint       # Lint code
```

## Architecture

Brief description of project structure based on analysis.

### Key Directories
- `src/` - Source code
- `tests/` - Test files
- `docs/` - Documentation

## Project-Specific Rules

1. ESM imports require `.js` extension: `import { x } from './y.js'`
2. Use Zod for all external input validation
3. Tests use vitest with `*.test.ts` naming convention

## Development Tips

- Run single test: `pnpm test -- path/to/test.ts`
- Monorepo: use `pnpm --filter <pkg>` for specific packages
```

#### 1.2 Secondary Output: `.ax/analysis.json`

**Location:** `.ax/analysis.json`
**Commit status:** Gitignored (regeneratable)
**Purpose:** Tool-accessible deep analysis for Tier 3/4

**Content:** Full `ProjectInfo` object from `ProjectAnalyzer` including:
- `architecture` (Tier 3)
- `security` (Tier 4)
- `moduleMap`
- `keyAbstractions`
- `publicAPI`

#### 1.3 Configuration: `.ax/config.json`

**Location:** `.ax/config.json`
**Commit status:** Optional (user preference)
**Purpose:** Provider-agnostic project settings

**Schema:**
```json
{
  "schemaVersion": "1.0",
  "defaultDepth": "standard",
  "language": "en",
  "customRules": []
}
```

### 2. Command Interface

#### 2.1 Basic Usage

```bash
# Default: generate AX.md with standard depth
ax init

# Or with ax-grok
ax-grok init
```

#### 2.2 Depth Control

| Flag | Tiers | Output Size | Use Case |
|------|-------|-------------|----------|
| `--depth=basic` | 1 | ~300 tokens | Quick setup, small projects |
| `--depth=standard` | 1-2 | ~600 tokens | Default, most projects |
| `--depth=full` | 1-3 | ~1000 tokens | Complex projects needing architecture context |
| `--depth=security` | 1-4 | ~1500 tokens | Security-sensitive projects |

#### 2.3 Additional Flags

| Flag | Description |
|------|-------------|
| `--output=<file>` | Custom output filename (default: `AX.md`) |
| `--refresh` | Update existing file with fresh analysis |
| `--migrate` | Convert from old 3-file format |
| `--force` | Overwrite existing files without prompting |
| `--dry-run` | Show what would be generated without writing |
| `--verbose` | Show detailed analysis progress |

#### 2.4 Future Flags (Phase 3)

| Flag | Description |
|------|-------------|
| `--web-search` | Enrich with real-time dependency/security checks |
| `--lang=<locale>` | Generate in specified language |
| `--interactive` | Post-generation refinement prompts |

### 3. Content Generation Rules

#### 3.1 Dynamic Path Detection

**Required behavior:**
- Scan for existing directories: `docs/`, `specs/`, `wiki/`, `.github/`
- Parse `package.json` scripts, `Makefile` targets, `pyproject.toml`
- Detect test directories: `tests/`, `test/`, `__tests__/`, `spec/`
- Never assume or impose organizational conventions

**Implementation:**
- Reuse existing `ProjectAnalyzer.detectDirectories()`
- Remove hardcoded paths from `LLMOptimizedInstructionGenerator`

#### 3.2 Ingest Existing Rules

**Parse these files if present:**
- `.cursorrules` - Cursor AI rules
- `.editorconfig` - Editor configuration
- `.eslintrc.*` - ESLint rules
- `.prettierrc.*` - Prettier rules
- `AX.md` - Existing AX instructions (for refresh)

**Behavior:**
- Extract relevant coding standards
- Avoid duplication with detected conventions
- Merge into "Project-Specific Rules" section

#### 3.3 Project-Specific Over Generic

**Remove generic advice:**
```markdown
<!-- ❌ REMOVE these generic rules -->
- DON'T: Introduce security vulnerabilities
- DON'T: Write untested code
- DO: Follow best practices
```

**Replace with project-specific rules derived from analysis:**
```markdown
<!-- ✅ KEEP project-specific rules -->
- This project uses pnpm workspaces—run commands from repo root
- ESM requires .js extension in imports
- Tests use vitest with *.test.ts naming; run `pnpm test`
- API keys stored encrypted in ~/.ax-{cli}/config.json
```

#### 3.4 Adaptive Verbosity

| Project Size | File Count | Default Tokens |
|--------------|------------|----------------|
| Small | < 20 files | ~300 |
| Medium | 20-100 files | ~600 |
| Large | 100+ files | ~1000 |

**Implementation:**
- Count source files during analysis
- Adjust section verbosity based on complexity
- Always respect explicit `--depth` flag over auto-detection

### 4. Migration Support

#### 4.1 Detection of Old Format

Check for presence of:
- `.ax-grok/CUSTOM.md` or `.ax-cli/CUSTOM.md`
- `ax.index.json` at project root
- `ax.summary.json` at project root

#### 4.2 Migration Command

```bash
ax init --migrate
```

**Behavior:**
1. Detect existing files
2. Extract customizations from `CUSTOM.md`
3. Generate new `AX.md` with merged content
4. Move deep analysis to `.ax/analysis.json`
5. Prompt to delete old files (with `--force` to skip prompt)
6. Output summary of changes

#### 4.3 Backward Compatibility

**Transition period (6 months):**
- `ax init --legacy` generates old 3-file format
- Deprecation warning when old format detected
- Documentation updated with migration guide

### 5. Provider Unification

#### 5.1 Directory Structure

**Old (remove):**
```
.ax-grok/
  CUSTOM.md
  settings.json
.ax-cli/
  CUSTOM.md
  settings.json
```

**New:**
```
.ax/
  analysis.json    # Deep analysis (gitignored)
  config.json      # Provider-agnostic settings
```

#### 5.2 Provider Hints

Instead of separate directories, use inline comments in `AX.md`:
```markdown
<!--
Provider notes:
- ax-grok: Supports web search via xAI API
- ax-cli: Local inference via Ollama/LMStudio
-->
```

#### 5.3 Config Migration

Merge provider-specific settings into unified `.ax/config.json`:
```json
{
  "schemaVersion": "1.0",
  "providers": {
    "ax-grok": { "webSearch": true },
    "ax-cli": { "localModel": "qwen3:14b" }
  }
}
```
